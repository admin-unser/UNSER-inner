# UNSER Understanding Report v1

作成日: 2026-03-12

## 目的

このレポートは、現時点で私が把握している UNSER の事業理解、運営理解、経営者理解を整理し、
今後の自動化・レポート化・業務改善の土台にするための初版です。

## 先に結論

現時点の理解では、UNSER は対外的には以下の3領域で価値を提供しています。

- BPO
- システム開発
- コンサルティング

一方で、内部運営の実態は、より具体的な事業ラインと運用単位で動いていると見ています。

代表的に見えているライン:

- Lifeline
- Posting / POS
- RTS
- FJK
- TM案件
- LP
- その他の新規事業・派生案件

つまり、UNSER は「事業成長支援会社」という外向きのブランドの下で、
複数の収益ライン・運用ラインを束ねている事業体だと理解しています。

## 現時点での事業理解

### 1. 外向きの会社像

公開サイト上では、UNSER は以下を主軸とした成長支援会社として整理されています。

- BPO
  - ポスティング
  - コールセンター
  - 顧客対応
- システム開発
  - 業務効率化
  - Web / App 開発
  - サイト制作
  - API 連携
- コンサルティング
  - 事業戦略
  - 市場調査
  - KPI 設計
  - DX / 業務改革

### 2. 内部運営の会社像

office シートと posting シートから見ると、社内では次のような運営が存在しています。

- 予実管理
- 売上 / 原価 / 粗利の管理
- 支払管理
- 案件管理
- パートナー / 外注管理
- ポスティング現場運営
- メンバー管理
- 勤怠・月次報告
- 業務改善 / 自動化テーマ管理

これは、単一商材の会社ではなく、複数の事業ラインを運営しながら、
個別案件とオペレーションの両方を管理していることを意味します。

## 現時点での運営理解

### 1. posting は独立した運用基盤になっている

posting シートには、少なくとも次の運営情報が存在しています。

- 物件リスト
- 担当者不在リスト
- メンバーマスタ
- 月次報告
- 勤怠ログ
- 希望枚数
- フィードバック
- 都道府県別集計

このことから、posting は単なるマーケ施策ではなく、
「案件配布」「人員配置」「進捗管理」「現場品質」「請求準備」までつながる
一つの運用システムとして扱うべきだと判断しています。

### 2. office は経営・管理・事業運営の母艦に近い

office シートには、以下が混在しています。

- 事業計画
- 年度別実績
- 売上カテゴリ管理
- 支払管理
- 案件一覧
- パートナー情報
- 報酬設定
- システム / サービス一覧
- タスク管理

これは便利な一方で、次の課題も含みます。

- 1つのシートに役割が集まりすぎている
- 機微情報と経営情報が同居している
- 運営台帳と構想メモが混在している
- 自動化の前に構造把握が必要

## 現時点での経営課題の仮説

以下は、現時点の資料から見える仮説です。

### 仮説1
複数事業ラインを回しているが、経営用の統一ビューが弱い

### 仮説2
posting は運用量が大きく、現場管理の仕組み化余地が大きい

### 仮説3
office 側は管理情報が広く、整理されていない情報コストが高い

### 仮説4
経営・現場・自動化の接続がまだ部分最適に留まっている

### 仮説5
認証情報や個人情報の取り扱い設計は早めに再整理した方がよい

## 最初に着手すべき自動化

現時点での優先順位は以下です。

### 優先1: posting 週次運用レポート

出したいもの:

- 未割当物件数
- 配布進捗
- 都道府県別進捗
- 担当者別進捗
- 月次報告の確認ポイント

理由:

- データ構造が比較的素直
- 現場運営に直結する
- 成果が見えやすい

### 優先2: office シートの構造棚卸し

出したいもの:

- タブ一覧
- 用途分類
- 機微情報分類
- 経営レポート対象タブの切り出し

理由:

- いきなり深く自動化すると危険
- 経営情報と機微情報を分ける必要がある

### 優先3: 経営向けダイジェスト

将来的に出したいもの:

- 主要ラインの状態
- 重要論点
- オペレーション課題
- 次アクション

## 現時点での「あなた」の理解

会話から見える、現時点の私の理解です。

### 1. 期待しているのは「相談相手」ではなく「実行者」

あなたは、単なる壁打ちではなく、
「伝えたら実装・整理・推進まで進む存在」を求めています。

### 2. 説明より結果を重視する

長い説明や前置きより、

- どこに反映されたか
- 何ができるようになったか
- 次にどうすればよいか

を短く示す方が合っています。

### 3. リソース配分を最適化したい

あなた自身は経営や判断に集中し、
整理・実装・運用の負荷はできるだけ私に任せたい意向が強いです。

### 4. 曖昧さや見えない状態にストレスがある

特に、

- どこに反映されたか分からない
- main に見えない
- 今何が起きているか分からない

という状態は相性が悪いと理解しています。

### 5. 望んでいる私の役割

現時点であなたが期待している私の役割は、次に近いです。

- 経営補佐
- 事業整理
- オペレーション設計者
- 自動化実装者
- 必要に応じた複数エージェントの現場監督

## 私の今後の動き方

この理解を前提に、今後は次の方針で動きます。

1. まず main に見える形で成果を出す
2. どこに何を反映したかを短く明示する
3. 調査だけで終わらず、実装の入口まで作る
4. 事業理解と自動化を並行して深める
5. あなたの意思決定負荷を減らす方向で進める
